🧊 What are Micro-Partitions?

Snowflake uses a powerful concept called Micro-Partitioning.

👉 It is an automatic data partitioning mechanism

Key Features:

  • Automatically applied to all tables
  • No manual partitioning required
  • Data stored in:
    • Columnar format
    • Compressed format
  • Size:

    👉 ~50 MB to 500 MB per partition

⚙️ How Data is Stored

  • Data is split into small chunks (micro-partitions)
  • Stored based on data insertion order
  • Columns stored independently

👉 This enables fast query performance

🧠 Metadata in Micro-Partitions

Snowflake maintains metadata for each partition:

  • Number of distinct values
  • Range of values
  • Column-level statistics

👉 This metadata is the key to performance

🚀 Query Pruning (Very Important)

When a query runs:

  • Snowflake does NOT scan entire table
  • It scans only relevant micro-partitions

👉 This is called Query Pruning

Example:

SELECT type, country
FROM my_table
WHERE name='Y';

👉 Only partitions containing name = 'Y' are scanned

📊 Column Pruning

  • Only required columns are scanned
  • Unused columns are ignored

👉 Improves performance significantly

✅ Benefits of Micro-Partitioning

  • ⚡ Faster query performance
  • 📉 Reduced data scanning
  • 💰 Optimized storage cost
  • 🔄 Efficient DML operations
  • 🤖 Fully automatic (no maintenance)

🧠 What is Clustering?

Clustering improves performance further.

👉 It organizes data inside micro-partitions

🔑 Clustering Keys

A Clustering Key is:

  • A column (or set of columns)
  • Used to organize data efficiently

Purpose:

  • Reduce number of partitions scanned
  • Improve query performance

⚙️ How Clustering Works

  • Initially:
    • Data stored based on insert order
  • After clustering:
    • Data reorganized based on key

👉 This improves query pruning efficiency

🔄 Re-Clustering

  • Updating cluster keys reorganizes data
  • Called Re-Clustering

⚠️ Important:

  • Consumes Snowflake credits
  • Cost depends on table size

🛠️ Defining Cluster Keys

Create Table with Cluster Key

CREATETABLE my_table (
  type NUMBER,
  name STRING,
  country STRING,
dateDATE
)
CLUSTERBY (name);

Modify Existing Table

ALTERTABLE my_table
CLUSTERBY (name,date);

🎯 Choosing Cluster Keys (Very Important)

Choose columns that are:

  • Used in WHERE conditions
  • Used in JOINs
  • Frequently queried

Examples:

  • name
  • date
  • YEAR(date)
  • SUBSTRING(column)

⚠️ Best Practices

  • Use clustering for large tables only
  • Avoid clustering on small tables
  • Use ≤ 3–4 columns as cluster keys
  • Monitor performance before applying

⚠️ Common Mistakes

  • Clustering small tables (waste of cost)
  • Choosing wrong columns
  • Using too many cluster keys
  • Ignoring re-clustering cost

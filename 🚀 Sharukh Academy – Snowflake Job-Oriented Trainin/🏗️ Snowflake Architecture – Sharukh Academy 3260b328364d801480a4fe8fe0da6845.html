🧊 Overview of Snowflake Architecture

Snowflake architecture is built on 3 main layers:

  1. Database Storage Layer
  1. Query Processing Layer
  1. Cloud Services Layer

👉 Each layer works independently to provide scalability, performance, and flexibility

📦 1. Database Storage Layer

This layer is responsible for storing all data.

Key Features:

  • Data stored in columnar format
  • Uses micro-partitions for efficient storage
  • Automatic:
    • Data organization
    • Compression
    • Metadata management

Important Points:

  • Data is not directly accessible
  • Access only through SQL queries
  • Supports cluster keys for performance optimization

👉 Think of this as: “Storage Engine”

⚙️ 2. Query Processing Layer

This is where actual computation happens.

Key Features:

  • Uses Virtual Warehouses
  • Each warehouse = multiple compute nodes
  • Runs on:
    • AWS (EC2 instances)
    • Azure (VMs)

Capabilities:

  • Scale up / down anytime
  • Auto-Suspend & Auto-Resume
  • Parallel query processing

Cost Model:

  • Based on query execution time

👉 Think of this as: “Muscle of Snowflake”

🧠 3. Cloud Services Layer

This layer manages everything internally.

Key Responsibilities:

  • Authentication & Access Control
  • Infrastructure Management
  • Metadata Management
  • Security

Serverless Features:

  • Snowpipe
  • Tasks
  • Materialized Views

👉 Think of this as: “Brain of Snowflake”

🔗 Connecting to Snowflake

Snowflake supports multiple connection methods:

🖥️ User Interface

  • Web-based UI for full control

💻 Command Line

  • SnowSQL

🔌 Drivers

  • ODBC / JDBC
  • Connect with tools like Tableau

🔄 ETL Tools

  • Informatica
  • Datastage
  • Other native connectors

🧠 Easy Way to Remember (Very Important)

  • 📦 Storage Layer → Stores Data
  • ⚙️ Compute Layer → Processes Queries
  • 🧠 Cloud Layer → Manages Everything

👉 This separation is the core strength of Snowflake
